🧠 Concepts fondamentaux

DADOU est un assistant IA personnel qui fonctionne localement sur votre machine. Contrairement aux IA cloud, il :

  • Se souvient — mémoire persistante de vos projets, préférences et décisions
  • Apprend — s'améliore session après session grâce à la rétroaction
  • Exécute — pilote des logiciels, écrit du code, manipule des fichiers
  • Protège — Guardian 3 niveaux valide chaque action avant exécution
ComposantRôleTechnologie
CoreLogique métier, ML, sécuritéRust (Axum, tokio, rusqlite)
FrontendInterface utilisateurReact 19 + Tauri v2
DashboardObservabilité temps réelHTML/JS autonome sur :7790
CLIAdministration, debugRust (clap)

🚀 Démarrage rapide

# 0. Choisir le dossier workspace (optionnel) $env:OPENHUMAN_WORKSPACE = "D:\MesProjets\DADOU" # 1. Lancer le core (terminal) .\target\release\dadou-core.exe serve # 2. Ouvrir le dashboard # → http://127.0.0.1:7790 # 3. Vérifier la santé curl http://127.0.0.1:7788/health # 4. Lister les skills installées .\target\release\dadou-core.exe skill list
Premier lancement : DADOU crée automatiquement le workspace dans %USERPROFILE%\DADOU (modifiable avec $env:OPENHUMAN_WORKSPACE) avec sa configuration, sa base SQLite et ses skills. Le mode offline est activé par défaut — aucun backend cloud requis.

💻 Cas 1 — Développeur : Analyse et refactoring d'un projet

Contexte : Vous reprenez un projet legacy de 50 000 lignes de Python. Vous devez comprendre l'architecture, identifier le code mort, et planifier le refactoring.

🔴 Session type — Premier contact avec le projet

1. DADOU indexe le projet

# Placez-vous dans le répertoire du projet cd C:\Projects\legacy-app # Demandez à DADOU d'analyser la structure # (via le chat Tauri) Analyse la structure de ce projet. Quels sont les modules principaux ? Quelles sont les dépendances critiques ? Y a-t-il du code mort ?

2. DADOU répond avec une cartographie

DADOU parcourt l'arborescence, lit les imports, identifie les points d'entrée, et stocke cette connaissance dans sa mémoire persistante. Il détecte :

Projet legacy-app — 47 modules Python, 12 340 lignes ├── core/ (12 modules) — Logique métier critique ├── api/ (8 modules) — Endpoints Flask, 3 non utilisés ⚠ ├── utils/ (15 modules) — 40% de code mort détecté ⚠ ├── migrations/ (7 modules) — Alembic, OK └── tests/ (5 modules) — Couverture 23% ❌ Top risques : pas de gestion d'erreurs dans api/, pas de rate limiting, SQL injection potentielle dans core/queries.py:142

3. DADOU se souvient entre les sessions

Le lendemain, vous rouvrez DADOU. Il restaure le contexte :

→ Reprise de session : projet legacy-app, Phase analyse (hier 18:42) → 3 faits projet en mémoire : structure, risques, code mort → Dernière action : review de core/queries.py → Continuer l'analyse ou passer au refactoring ?

4. Refactoring assisté

# DADOU propose un plan de refactoring → Plan : Extraire api/ en microservice indépendant 1. Créer api-v2/ avec FastAPI 2. Migrer les endpoints un par un (8 endpoints, 3 actifs) 3. Ajouter rate limiting (Guardian valide les appels réseau) 4. Tests unitaires pour chaque endpoint migré
Le Guardian protège : quand DADOU propose de modifier core/queries.py, le Guardian N1 vérifie que le fichier est dans le workspace, N2 scanne les patterns d'exfiltration, et N3 valide l'intention de la modification.

📊 Cas 2 — Data Scientist : Pipeline d'analyse de données

Contexte : Vous devez analyser un jeu de données CSV de 2 GB, créer des visualisations, et générer un rapport.

🔴 Session type — Pipeline data

1. Installation d'un skill Python

# Installer un skill d'analyse de données dadou-core skill install https://github.com/dadou/skill-pandas.git # Vérifier l'installation dadou-core skill list → pandas-analyzer v1.2.0 [enabled] GPG ✓ runtime: python

2. Analyse interactive

Analyse ce fichier CSV : C:\Data\ventes-2025.csv → Colonnes détectées : date, produit, quantité, prix, région → 1 247 892 lignes, 0 valeurs nulles → Période : 2025-01-01 → 2025-12-31 Statistiques descriptives : - CA total : 14 230 450 € - Top région : Île-de-France (34% des ventes) - Produit phare : REF-8842 (12 450 unités) - Tendance : +8% vs 2024

3. Génération de rapport

Génère un rapport PDF avec les visualisations clés. → Rapport généré : C:\Data\rapport-ventes-2025.pdf (14 pages) → Contenu : résumé exécutif, 6 graphiques, recommandations → Fichiers source : analyse.py, data_cleaned.csv, charts/
Attention : l'exécution Python passe par le sidecar Docker (si installé). Sans Docker, le fallback local est utilisé — moins isolé mais fonctionnel. Docker Desktop est recommandé pour les données sensibles.

🔧 Cas 3 — Maker : Automatisation domotique

Contexte : Vous gérez une maison connectée avec Home Assistant, des capteurs ESP32 et des scripts d'automatisation.

🔴 Session type — Automatisation

1. Création d'un script d'automatisation

Crée un script Python qui : - Lit la température du capteur ESP32 (MQTT) - Si température > 26°C, allume la clim via Home Assistant - Si température < 19°C, allume le chauffage - Log toutes les actions dans un fichier CSV → Script créé : C:\Projects\domotique\temp_control.py → Testé avec des valeurs simulées → Déploiement : cron toutes les 5 minutes

2. DADOU gère le cron

# Configurer une tâche périodique dadou-core cron add temp-check "*/5 * * * *" "python temp_control.py" # Voir l'historique d'exécution dadou-core cron history → temp-check | 14:05 | OK (340ms) | Temp: 24°C, Aucune action → temp-check | 14:10 | OK (320ms) | Temp: 27°C, Clim allumée → temp-check | 14:15 | OK (310ms) | Temp: 25°C, Clim éteinte

3. Dashboard temps réel

Ouvrez http://127.0.0.1:7790 pour voir :

  • Le Guardian qui valide chaque appel MQTT (N1: allow, N2: allow, N3: skip)
  • Les skills actives (mqtt-client, home-assistant)
  • L'historique des actions (354 exécutions, 0 erreur)
  • La consommation mémoire (127 MB, stable)

⚖ Cas 4 — Juriste : Analyse de contrats

Contexte : Vous devez analyser un contrat de 40 pages, identifier les clauses sensibles, et comparer avec un contrat de référence.

🔴 Session type — Analyse juridique

1. DADOU lit et structure

Analyse ce contrat : C:\Contrats\Partenariat-2025.pdf → Pages : 42 | Type : Partenariat commercial → Parties : Société A (France) ↔ Société B (Allemagne) → Droit applicable : Français Clauses identifiées : Art 4.2 — Clause de non-concurrence (24 mois, France) Art 7.1 — Responsabilité limitée à 50 000 € Art 12.3 — Résiliation unilatérale (préavis 90 jours) Art 15.0 — Attribution de juridiction (Tribunal de Commerce de Paris)

2. Détection des clauses à risque

Compare avec le contrat-type "Partenariat Standard v3". → 3 divergences détectées : ⚠ Art 4.2 : Non-concurrence 24 mois (standard : 12 mois) ⚠ Art 7.1 : Plafond de responsabilité 50 K€ (standard : 200 K€) ⚠ Art 15.0 : Juridiction Paris (standard : arbitrage CCI) → Recommandation : renégocier Art 4.2 et 7.1 avant signature
Confidentialité : Toute l'analyse se fait localement. Les documents ne quittent jamais votre machine. Le Guardian bloque toute tentative d'exfiltration réseau. Les fichiers sont stockés avec provenance et contrôle d'accès.

(2024) — "Impact du COVID-19 sur l'emploi en Côte d'Ivoire" → Résumé généré pour chaque source (200 mots) → BibTeX généré automatiquement

🧩 Installation et utilisation des Skills

Les skills sont des modules externes (WASM ou Python) que DADOU peut installer depuis Git.

Installer un skill

# Depuis un repo Git (WASM) dadou-core skill install https://github.com/dadou/skill-filesystem.git # Depuis un repo Git (Python) dadou-core skill install https://github.com/dadou/skill-pandas.git # Lister les skills installées dadou-core skill list # Mettre à jour dadou-core skill update skill-filesystem # Auditer (ré-analyse statique) dadou-core skill audit skill-filesystem # Désinstaller dadou-core skill remove skill-filesystem

Structure d'un skill (dadou-skill.yaml)

# Skill WASM name: mon-skill version: 0.1.0 wasm: path: build/output.wasm entry: run permissions: network: false filesystem: read: ["/tmp/**"] # Skill Python name: mon-skill-python version: 0.2.0 python: entry: main.py requirements: - pandas>=2.0 - requests runtime: docker permissions: network: true

📊 Le Dashboard d'observabilité

Accessible sur http://127.0.0.1:7790, le dashboard affiche en temps réel :

Vue Guardian

Chaque action de DADOU est évaluée par le Guardian à 3 niveaux :

NiveauTypeLatenceRôle
N1Règles déterministes<1msBloque chemins interdits, patterns malveillants
N2Heuristiques (entropie, exfiltration)<10msDétecte patterns suspects, escalade si doute
N3LLM validator<500msValide l'intention des actions ambiguës
Exemple — Timeline Guardian : 14:05:23 | N1 ALLOW | file_read | path within workspace 14:05:23 | N2 ALLOW | file_read | entropy: 0.12 (safe) 14:05:24 | N1 ALLOW | shell | command: git status 14:05:24 | N2 ALLOW | shell | no exfil pattern detected 14:05:25 | N1 BLOCK | file_write| path: C:\Windows\System32 ❌ 14:05:25 | N2 — | — | (N1 already blocked, early exit)

Semantic Router

Le dashboard permet de tester la découverte de skills par similarité sémantique :

> Requête : "analyser des données CSV" → git-helper 0.23 (keyword) → pandas-runner 0.78 (cosine) → web-scraper 0.12 (keyword)

🔧 Dépannage

Problèmes courants

Le core ne démarre pas (port 7788 occupé)

Changez le port dans la config :

set OPENHUMAN_CORE_PORT=7799 dadou-core serve
Le dashboard n'affiche rien (port 7790)

Vérifiez que le dashboard est activé :

# config.toml [dashboard] enabled = true port = 7790

Si le port 7790 est occupé, DADOU essaie automatiquement 7791-7794.

Erreur "cmake not found" lors du build
# Vérifier cmake cmake --version # Installer si absent winget install Kitware.CMake
Erreur "libclang.dll not found"
# PowerShell (admin) winget install LLVM.LLVM [System.Environment]::SetEnvironmentVariable("LIBCLANG_PATH","C:\Program Files\LLVM\bin","User")
Docker non disponible pour les skills Python

DADOU bascule automatiquement sur Python local. Un avertissement est loggé :

[python-skill] Docker is not installed. Python skills will run using your local Python interpreter, which has less isolation.
Le Guardian bloque une action légitime

Vérifiez les logs et ajustez la config :

# Voir les décisions Guardian dans le dashboard → http://127.0.0.1:7790 # Ou via les logs type ~\.openhuman\workspace\dashboard\dashboard.db sqlite3 ~\.openhuman\workspace\dashboard\dashboard.db \ "SELECT * FROM dashboard_events WHERE kind='guardian_blocked' ORDER BY recorded_at DESC LIMIT 10"
