<objective>
Validate every skill output semantically before reinjecting it into the LLM conversation, using deterministic pattern rules plus an optional LLM deep-check.

Purpose: Even with structured JSON envelopes (Plan 05-02), skill output could contain prompt injection attempts. Semantic validation adds a second layer of defense: rule-based detection catches known patterns (instruction overrides, system prompt manipulation), and an optional LLM check catches novel or obfuscated attempts.

Output:
- `SemanticOutputValidator` with rule-based + LLM-based validation
- Injection pattern rule set (15+ patterns)
- Tool loop integration point
- JSON-RPC controllers for querying validation results
</objective>

       ```rust
       pub mod rules;
       pub mod llm_check;
       
       use self::rules::{check_injection_patterns, InjectionFinding};
       use self::llm_check::llm_deep_check;
       
       /// How strictly the validator treats suspicious output.
       #[derive(Debug, Clone, Copy, PartialEq, Eq)]
       pub enum ValidationMode {
           /// Block output if any rule triggers (fail-closed). Default.
           Strict,
           /// Warn and tag output as suspicious but allow through.
           Relaxed,
       }
       
       /// Configuration for the semantic validator.
       #[derive(Debug, Clone)]
       pub struct ValidatorConfig {
           pub mode: ValidationMode,
           /// Whether to perform LLM deep-check on rule suspicion.
           pub enable_llm_check: bool,
           /// Max characters to analyze (to bound LLM call cost).
           pub max_analysis_chars: usize,
       }
       
       impl Default for ValidatorConfig {
           fn default() -> Self {
               Self {
                   mode: ValidationMode::Strict,
                   enable_llm_check: false, // v1: rules-only by default
                   max_analysis_chars: 10_000,
               }
           }
       }
       
       /// Result of semantic validation.
       #[derive(Debug, Clone, serde::Serialize)]
       pub struct ValidationResult {
           /// Whether the output passed validation.
           pub allowed: bool,
           /// Findings from rule-based checks.
           pub rule_findings: Vec<InjectionFinding>,
           /// Result of LLM deep check (if performed).
           pub llm_verdict: Option<LlmVerdict>,
           /// Human-readable summary of the validation decision.
           pub summary: String,
       }
       
       /// LLM verdict on the output (placeholder — full type in llm_check.rs).
       #[derive(Debug, Clone, serde::Serialize)]
       pub enum LlmVerdict { Safe, Suspicious, Malicious }
       
       /// Semantic output validator — validates skill output before LLM reinjection.
       pub struct SemanticOutputValidator {
           config: ValidatorConfig,
       }
       
       impl SemanticOutputValidator {
           pub fn new(config: ValidatorConfig) -> Self { Self { config } }
           
           /// Validate skill output text against injection patterns.
           pub fn validate(&self, skill_name: &str, output: &str) -> ValidationResult {
               let analysis = &output[..output.len().min(self.config.max_analysis_chars)];
               let rule_findings = check_injection_patterns(analysis);
               
               let blocked = if self.config.mode == ValidationMode::Strict {
                   !rule_findings.is_empty()
               } else {
                   false
               };
               
               let summary = if rule_findings.is_empty() {
                   format!("[anti-injection] skill '{}' output passed validation", skill_name)
               } else if blocked {
                   format!(
                       "[anti-injection] BLOCKED skill '{}' output: {} rule(s) triggered",
                       skill_name, rule_findings.len()
                   )
               } else {
                   format!(
                       "[anti-injection] WARN skill '{}' output: {} rule(s) triggered (relaxed mode)",
                       skill_name, rule_findings.len()
                   )
               };
               
               ValidationResult { allowed: !blocked, rule_findings, llm_verdict: None, summary }
           }
       }
       ```

    4. Create `src/openhuman/anti_injection/validator/rules.rs` with 15+ injection detection patterns:

       Define:
       ```rust
       #[derive(Debug, Clone, serde::Serialize)]
       pub struct InjectionFinding {
           pub rule_name: String,
           pub severity: FindingSeverity,
           pub matched_text: String,
           pub position: usize, // byte offset of match
       }
       
       #[derive(Debug, Clone, Copy, PartialEq, Eq, serde::Serialize)]
       pub enum FindingSeverity { High, Medium, Low }
       
       /// Check `content` against all known injection patterns.
       /// Returns a list of findings (empty = no injection detected).
       pub fn check_injection_patterns(content: &str) -> Vec<InjectionFinding> {
           let mut findings = Vec::new();
           for rule in ALL_INJECTION_RULES {
               if let Some(finding) = rule.evaluate(content) {
                   findings.push(finding);
               }
           }
           findings
       }
       ```

       Implement `ALL_INJECTION_RULES` as a static array of `InjectionRule` structs, each with:
       - A regex pattern (use `regex::Regex`)
       - A severity
       - A name

       REQUIRED patterns (at least 15):

       | # | Rule Name | Pattern Targets | Severity |

       | 1 | `ignore_previous_instructions` | "ignore (all )?(previous|above|prior) (instructions|directives|commands)" | High |
       | 2 | `system_prompt_override` | "system prompt", "you are now", "new instructions" | High |
       | 3 | `role_switch` | "pretend (you are|to be)", "act as if", "from now on you are" | High |
       | 4 | `tool_abuse` | "use the .* tool to", "call the .* function" (in output context) | High |
       | 5 | `output_format_injection` | "output format", "respond in", "reply in the following format" (instructional) | Medium |
       | 6 | `data_exfiltration_request` | "send (this|the) (data|information|content) to", "post to", "upload to" | High |
       | 7 | `reverse_injection` | "ignore the previous system prompt", "you must now" | High |
       | 8 | `hidden_base64` | Base64-encoded string > 40 chars (detected via regex `[A-Za-z0-9+/]{40,}={0,2}`) | Medium |
       | 9 | `hidden_hex` | Long hex string suggesting encoded payload (`[0-9a-f]{40,}`) | Low |
       | 10 | `markdown_injection` | Embedded markdown that could reshape the prompt (`###`, `## System`) | Medium |
       | 11 | `url_injection` | Instruction to visit a URL and act on its content | Medium |

       | 15 | `chain_injection` | Multi-step instructions that chain tools for injection | Medium |
       | 16 | `xml_tag_abuse` | Closing or forging of `<external_data>` or `<tool_result>` tags to escape wrapping | High |
