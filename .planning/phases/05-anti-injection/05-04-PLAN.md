<objective>
Require the LLM to emit structured JSON action plans, then validate the complete plan through the Guardian pipeline before any step executes.

Purpose: The most powerful prompt injection attacks are multi-step: they don't just suggest a single tool call, they describe a sequence of actions that subvert the agent's purpose. By forcing the LLM to declare its full plan upfront and validating it through the Guardian's layered pipeline (N1 rules + N2 heuristics + N3 LLM), we catch injection attacks at the planning stage rather than reacting to each tool call individually.

Output:
- `StructuredPlan` and `PlanValidationResult` types
- `GuardianPipeline::evaluate_plan()` method
- `ExecutionProtocolSection` in system prompt
- Plan interception in tool loop
- Plan validation DomainEvents
</objective>

       ```rust
       use super::types::{StructuredPlan, PlanValidationResult};
       
       impl GuardianPipeline {
           /// Evaluate a structured action plan through the full Guardian pipeline.
           ///
           /// Validation flow:
           /// 1. **Structure check** — Verify the plan JSON has all required fields
           /// 2. **Intent check** — Evaluate the overall goal for malicious intent (via N3)
           /// 3. **Step-by-step check** — Run each step through N1->N2->N3 pipeline
           ///
           /// Returns `PlanValidationResult` with `allowed: false` and `blocked_by`
           /// set to the first rejection level.
           pub async fn evaluate_plan(&self, plan: &StructuredPlan) -> PlanValidationResult {
               // ── Stage 1: Structure check ──
               if plan.goal.trim().is_empty() {
                   return PlanValidationResult {
                       allowed: false,
                       blocked_by: "structure".into(),
                       reasoning: "Plan goal is empty".into(),
                       rejected_steps: vec![],
                       step_results: vec![],
                   };
               }
               
               // Cap the plan at 20 steps to prevent abuse.
               const MAX_STEPS: usize = 20;
               if plan.steps.len() > MAX_STEPS {
                   return PlanValidationResult {
                       allowed: false,
                       blocked_by: "structure".into(),
                       reasoning: format!("Plan exceeds maximum step count ({}/{} step(s))",
                           plan.steps.len(), MAX_STEPS),
                       rejected_steps: (MAX_STEPS..plan.steps.len()).collect(),
                       step_results: vec![],
                   };
               }
               
               // ── Stage 2: Step-by-step validation via N1→N2→N3 pipeline ──
               let mut step_results = Vec::with_capacity(plan.steps.len());
               let mut rejected_steps = Vec::new();
               
               for (i, step) in plan.steps.iter().enumerate() {
                   let command = if step.tool == "bash" || step.tool == "shell" {
                       step.args.get("command").and_then(|v| v.as_str())
                   } else { None };
                   
                   let file_path = match step.tool.as_str() {
                       "file_write" | "edit" | "file_read" | "glob" | "grep"
                       | "list_files" | "glob_search" | "read_diff" => {
                           step.args.get("path").and_then(|v| v.as_str())
                       }
                       _ => None,
                   };
                   
                   let result = self.evaluate(&step.tool, &step.args, command, file_path).await;
                   step_results.push(result.clone());
                   
                   if !result.allowed {
                       rejected_steps.push(i);
                   }
               }
               
               // ── Stage 3: Final decision ──
               let all_allowed = rejected_steps.is_empty();
               let blocked_by = if rejected_steps.is_empty() {
                   "none".into()
               } else {
                   // Use the first rejection level
                   step_results[rejected_steps[0]].blocked_by.clone()
               };
               
               let reasoning = if all_allowed {
                   format!("Plan '{}' validated: {} step(s) all passed Guardian pipeline",
                       plan.goal.chars().take(80).collect::<String>(),
                       plan.steps.len())
               } else {
                   format!("Plan '{}' blocked at {}: step(s) {:?} rejected",
                       plan.goal.chars().take(80).collect::<String>(),
                       blocked_by, rejected_steps)
               };
               
               PlanValidationResult {
                   allowed: all_allowed,
                   blocked_by,
                   reasoning,
                   rejected_steps,
                   step_results,
               }
           }
       }
       ```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| LLM -> Plan emission | The LLM emits a structured JSON plan; untrusted content in the prompt could influence the plan |
| Plan -> Guardian validation | The Guardian validates the complete plan before any execution; the LLM cannot bypass validation |
| Per-step -> Tool execution | Individual steps are validated through the existing N1->N2->N3 pipeline |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-12 | Spoofing | extract_structured_plan | mitigate | Plan must be valid JSON matching StructuredPlan schema; malformed plans are rejected |
| T-05-13 | Tampering | evaluate_plan | mitigate | Each step is independently validated through N1->N2->N3; plan structure is capped at 20 steps |
| T-05-14 | Elevation of Privilege | Plan validation bypass | mitigate | Per-step Guardian validation still runs even after plan is accepted (defense in depth) |
| T-05-15 | Tampering | Plan extraction | accept | LLM could emit non-plan tool calls directly; plan validation is bonus, not exclusive path |
| T-05-SC | Tampering | packages | mitigate | Zero new dependencies — uses existing `regex` and `serde_json` |
</threat_model>
