    ```rust
    /// Score de suspicion d'un detecteur N2.
    pub struct N2Score {
        pub score: f64,           // 0.0 (safe) a 1.0 (malicious)
        pub reason: String,       // Pourquoi ce score
        pub triggered_by: String, // Nom du detecteur (ex: "exfiltration", "entropy", "hidden_payloads")
    }

    impl N2Score {
        pub fn is_blocking(&self, threshold: f64) -> bool {
            self.score >= threshold
        }
        pub fn is_escalating(&self, threshold: f64) -> bool {
            self.score >= threshold && self.score < BLOCK_THRESHOLD
        }
    }

    /// Resultat de l'evaluation N2 complete.
    pub struct N2Result {
        pub allowed: bool,        // false si block threshold depasse
        pub escalate: bool,       // true si escalate threshold depasse mais pas block
        pub scores: Vec<N2Score>, // Scores individuels des detecteurs
        pub latency_us: u64,      // Temps total en microsecondes
    }

    /// Configuration du moteur N2.
    pub struct N2EngineConfig {
        pub block_threshold: f64,     // defaut: 0.7
        pub escalate_threshold: f64,  // defaut: 0.3
        pub max_input_chars: usize,   // defaut: 10000 (limite d'analyse pour perf)
    }
    ```

    Dans `n2/mod.rs`, creer `GuardianN2` struct:

    ```rust
    pub struct GuardianN2 {
        config: N2EngineConfig,
        exfiltration_detector: ExfiltrationDetector,
        entropy_analyzer: EntropyAnalyzer,
        hidden_payloads_detector: HiddenPayloadsDetector,
    }

    impl GuardianN2 {
        pub fn new(config: N2EngineConfig) -> Self;
        pub fn with_defaults() -> Self; // N2EngineConfig::default()

        /// Execute les 3 detecteurs N2, combine les scores, et retourne N2Result.
        /// PRE-execution: analyse tool_name + arguments + command.
        /// Ne prend pas tool_output (POST-execution sera ajoute en Phase 5).
        pub fn evaluate(
            &self,
            tool_name: &str,
            tool_args: &serde_json::Value,
            command: Option<&str>,
            file_path: Option<&str>,
        ) -> N2Result;
    }
    ```

    La methode `evaluate()`:
    1. Mesure le temps avec `std::time::Instant`
    2. Construit une String d'entree depuis les args + command
    3. Execute les 3 detecteurs en sequence (ils sont synchrones, <10ms)
    4. Agrege les scores: si ANY score >= block_threshold -> allowed=false
    5. Si pas de block mais ANY score >= escalate_threshold -> escalate=true
    6. Retourne N2Result

    Les constructeurs `ExfiltrationDetector`, `EntropyAnalyzer`, `HiddenPayloadsDetector`
    sont definis dans leurs fichiers respectifs. Dans ce task, on declare les types mais
    les implementations concretes sont dans les Tasks 2-4.

    ```rust
    /// Analyse l'entropie des arguments outil et commandes.
    pub struct EntropyAnalyzer;

    impl EntropyAnalyzer {
        pub fn new() -> Self;

        /// Calcule l'entropie Shannon de la chaine.
        /// H(X) = -sum(p_i * log2(p_i)) pour chaque caractere unique.
        /// Retourne l'entropie en bits par caractere.
        pub fn shannon_entropy(data: &str) -> f64;

        /// Analyse les arguments et commande pour detecter les chaines a haute entropie.
        /// Decoupe en tokens (espaces, quotes), calcule l'entropie de chaque token.
        /// Si un token a entropie > threshold, retourne un N2Score.
        pub fn analyze(&self, args_str: &str, command: Option<&str>) -> Option<N2Score>;
    }
    ```

    Implementation de `shannon_entropy`:
    1. Compter la frequence de chaque caractere
    2. Calculer la probabilite: `p_i = count_i / total_chars`
    3. `H = -sum(p_i * log2(p_i))`
    4. Retourner l'entropie en bits par caractere

    La methode `analyze`:
    1. Tokenize la commande + args en tokens (split sur espaces, preserve les quotes)
    2. Pour chaque token de taille >= 8 caracteres, calculer l'entropie
    3. Mapping entropie -> score:
       - H < 4.0 -> score 0.0 (normal)
       - 4.0 <= H < 4.5 -> score 0.2 (legerement suspect)
       - 4.5 <= H < 5.5 -> score 0.4 (suspect)
       - 5.5 <= H < 6.0 -> score 0.7 (bloquant)
       - H >= 6.0 -> score 1.0 (tres bloquant)
    4. Si le score max est > 0.3, retourner un N2Score avec le token concerne

    Tests: utiliser des chaines de test connues. Exemple: texte anglais ~3.5 bits/car, base64 ~6.0, hex ~4.5.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Tool arguments -> N2 detectors | Arguments utilisateur analyses par regex et entropie. Donnees non fiables. |
| N2 scores -> pipeline decision | Les scores N2 peuvent etre contournes si un attaquant connait les patterns. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | N2 regex patterns | mitigate | Les patterns sont compiles en Rust, pas modifiables depuis YAML. Mise a jour via mise a jour du binaire. |
| T-03-02 | Information Disclosure | N2Score reasons | accept | Les raisons de blocage sont exposees via RPC. Pas de donnees sensibles (uniquement nom du pattern + extrait). |
| T-03-03 | Denial of Service | Entropy analysis on large input | mitigate | `max_input_chars` limite la taille analyse (defaut 10000). Troncature silencieuse. |
| T-03-04 | Elevation of Privilege | N2 bypass | accept | N2 est un filet de securite supplementaire. N1 est le niveau de confiance principal. Un bypass N2 est rattrape par N3 si escalade. |
| T-03-SC | Tampering | Cargo crate imports | mitigate | Aucun nouveau crate externe. Utilise `regex` (deja dans le workspace) et `std` only pour l'entropie. |
</threat_model>
